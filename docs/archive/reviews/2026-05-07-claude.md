## 4. 调研要点摘要（权威来源）

### Anthropic Skills 官方约束
- **SKILL.md ≤ 500 行硬上限**
- **Progressive disclosure 三级加载**：frontmatter（启动常驻）→ body（触发后读）→ references/scripts（按需）
- **只允许一层引用深度**：SKILL.md → reference.md ✓；reference.md → details.md ✗
- **description 写法**：第三人称 + "做什么 + Use when ..." + 关键词

### CLAUDE.md 共识（2026）
- **硬性 ≤ 200-300 行**；每轮注入，5000 token 整段会话都背着
- 启发式：`In this project...` 留 CLAUDE.md；`When doing X...` 进 skill

### Hooks 退出码语义
- **exit 2 + stderr** = block + Claude 重试
- **exit 1** = warning，**Claude 不会重试**
- **exit 0 + stdout** = LLM **看不见**（除特殊事件）
- 想让模型看到非阻塞反馈：JSON `{"hookSpecificOutput":{"additionalContext":"..."}}`

### AGENTS.md 实标准
- 60,000+ 仓库采用，24+ 工具兼容
- Claude Code 暂不原生读，社区做法是 AGENTS.md + CLAUDE.md 共存

### llms.txt 现状
- Google 公开表态："No AI system currently uses llms.txt"
- 单边发布姿态；对工作区项目实际收益接近零

### MCP 分发
- 官方 registry 仅接 npm public registry
- stdio = 本地；Streamable HTTP = 远程推荐

### 安装方式
- curl|bash 在企业审计场景受抑
- 国内通行：GHProxy / cnpmjs / Gitee 镜像

### 鸿蒙生态信号（2025-2026）
- 头号痛点：AI 生成 ArkTS 编译不过
- #2：提审被拒；#3：V1/V2 装饰器混用
- 调真机已不是 Top 痛点
- 鸿蒙 AI skill 项目 GitHub star 普遍 < 100，蓝海

## 5. v2 修订版优化清单

**v2 分级原则**：按"是否直接对应鸿蒙开发者真痛点"分层。

- **核心层**：直接解决头号痛点（AI 编译不过 / V1V2 混用 / 版本错配）
- **合规层**：解决工程合规与正确性（hook 退出码、skill 行数硬约束）
- **扩展层**：扩展用户面或装机体验（多工具支持、安装审计性）
- **体验层**：上手友好度
- **Backlog**：独立项目级工作量，不进当前计划

### 核心层 — 直接对应用户头号痛点（必做）

#### **C-1：自建 ArkTS 规范库 → arkts-rules skill 引用规范条款**（原 P1-3 升级）
**对应痛点**：#1 AI 生成 ArkTS 编译不过

**为什么真值得**：A 当前 arkts-rules 是手写规则，没有"规范源头"可追溯。AI 仍可能编造。引入官方规范条款引用机制后，每条 ARKTS-NNN 错误可对应规范条款 ID，AI 生成代码时被强制查规范。这是直接攻击头号痛点的唯一一条。

**具体动作**：
- arkts-rules skill 加 references/spec/ 子目录（一层内），分类索引 ArkTS 1.2.0+ 规范条款
- 每条 ARKTS-NNN 规则附"对应规范条款 + 链接"
- frontmatter description 加"必须引用规范条款，不能编造"
- hook 在 detect 到 arkts-no-* 错误时输出 additionalContext 引导 Claude 去 references/spec/ 查
- **先 sample 验证 E 的 OCR 是 ArkTS 1.2 还是 1.3+**；若 ≤ 1.2 则需自行从华为开发者联盟拉最新规范重 OCR

#### **C-2：state-management skill V2 装饰器审计**（原 P0-6）
**对应痛点**：#3 V1/V2 混用导致状态不刷新

**为什么真值得**：调研显示 V1/V2 混用是开发者第三高频痛点。在 < 500 行 SKILL.md 内同时讲透 V1（10 装饰器）+ V2（11 装饰器）+ 反模式 + 不混用规则几乎不可能，**很可能 V1 部分写得多、V2 部分写得薄**——而鸿蒙 6 时代 V2 是事实标准。

**具体动作**：
- 主 SKILL.md：触发判定 + V1/V2 选型决策树 + 5 条最高频反模式
- references/v1.md：V1 的 10 装饰器 + 历史代码维护场景
- references/v2.md：V2 的 11 装饰器（@Local/@Param/@Event/@Once/@Provider/@Consumer/@ObservedV2/@Trace/@Monitor/@Computed/@ComponentV2）+ 鸿蒙 6 推荐用法
- references/migration-v1-to-v2.md：迁移指南 + ID 化判定

#### **C-3：版本契约自检**（原 P0-5）
**对应痛点**：防止 A 自己变成下一个 G/H

**为什么真值得**：鸿蒙快速迭代是长期事实，没有版本契约 = 必然过期。G 和 H 的命运是反面教材。版本契约让用户和 Claude 都能即时判断"这套 skill 是否对齐当下鸿蒙版本"。这条是 A 区别于 B/C/D/E/F 的真正可持续壁垒——B/C/D/E/F 都没有显式版本契约。

**具体动作**：
- README + AGENTS.md 顶部加版本契约：
```yaml
targets:
  harmonyos: ">= 6.0.0 / API 18+"
  arkts: ">= 1.2.0"
  deveco_studio: ">= 5.1"
  last_verified: "2026-05-06"
```
- 5 个 skill frontmatter 加 `verified_against: harmonyos-6.0.0-api18`
- 新增 `tools/version-check.sh`：解析当前 oh-package.json5 的 compatibleSdkVersion，与 skill 声明对比，不匹配时 PostToolUse 输出 additionalContext
- CHANGELOG.md 按"鸿蒙版本 + 规则变更"组织

#### **C-4：PostToolUse hook 退出码核查**（原 P0-3）
**对应痛点**：核心闭环正确性——所有 hook 投入产出比的前提

**为什么真值得**：如果 A 现在的 hook 退出码用错了，等于"hook 跑了但 Claude 看不见 / 不重试"——所有 hook 投入沦为噪音。这是隐性 bug，是验证已有投资是否真生效的前置。

**具体动作**：
- 全量核查 `tools/hooks/post-edit.sh` 和 `lib/scan-arkts.sh` / `check-ohpm-deps.sh`：
  - ARKTS 强制规则违规 → exit 2 + stderr 写编号化错因（`[ARKTS-001] 检测到 any 类型`）
  - OHPM 未知降级 → exit 0 + stdout JSON `additionalContext`
  - P2 警告 → 同上 additionalContext，**不要 exit 1**
- 给所有 hook 命令加 `timeout 10`

- 拆 references/：保留触发判定 + 子任务索引在主 SKILL.md
- ARKTS 13 反模式细则、9 大 review 类目、20 个 AGC 拒因放进各自 references
- **只允许一层深**。100+ 行 reference 文件加 TOC

### 扩展层 — 工程化扩展（强建议）

#### **E-1：AGENTS.md 重新定位为"通用宪法 + 主源"**（原 P0-4 修订）
**对应价值**：扩展用户面（Codex/Cursor/Aider/Amp/Gemini CLI/Windsurf/Junie/Zed/...24+ 工具）

**v2 修订**：不强行 symlink（会让 CLAUDE.md 失去 Claude 特有的 skill 索引），而是：
- AGENTS.md = "通用宪法"，所有工具共享的硬约束
- CLAUDE.md = `@AGENTS.md` 引用 + Claude 特有的 skill 索引段
- install.sh 的 `--targets=cursor,copilot` 实质上变为"复制 AGENTS.md 到对应位置"

**为什么不是"为做而做"**：60K+ 仓库已采用 AGENTS.md，24+ 工具读它。A 投入很小（160 行 AGENTS.md 已存在）就能从 4 工具支持跳到 24+。注意：**这条对鸿蒙开发者头号痛点零关联**，但扩展用户面是合理的"次要价值"。

#### **E-2：稳定 ID 体系收口**（原 P1-4）
**对应价值**：工程化壁垒（A 已有 AGC-RJ-NNN，自然延伸）

**为什么真值得**：A 已经在 hook 里隐式用 ARKTS-001 / STATE-002 等。让 5 个 skill 都按 ARKTS-NNN / STATE-NNN / BUILD-NNN / SIGN-NNN / REVIEW-NNN 编号化、harmonyos-review 输出报告时直接引用 ID，是把已有投资正式化为工程标准。

**具体动作**：
- 一次定到位，rename = breaking change
- harmonyos-review skill 输出报告时引用 ID
- README + AGENTS.md 加"如何用 ID 系统反查规则"

#### **E-3：`.mcp.json` 改 `npx -y` 形式**（原 P1-2）
**对应价值**：fresh clone 一次跑通的基础可用性

**为什么真值得**：当前 `.mcp.json` 写死 `mcp-harmonyos` 命令，但 README 没说从哪装。fresh clone 用户必然卡住。

**具体动作**：
```json
{ "mcpServers": { "harmonyos": { "command": "npx", "args": ["-y", "@openharmony/mcp-harmonyos@latest"] } } }
```
README 写明镜像兜底。

#### **E-4：npx 安装入口 + 镜像默认推荐**（原 P0-7 + P1-1 合并）
**对应价值**：装机审计性 + 国内首装成功率

**为什么真值得**（合并的理由）：原 P0-7 和 P1-1 实质上都是"让首装更可靠"的同一件事——curl 黑盒（企业审计反对）+ 国内 GitHub 不通（rawguthub.com 直连失败率高）。一起改成"npx 入口 + 国内自动探测镜像"是最自然的组合。

**具体动作**：
- README 第一安装命令改为 `npx -y @harmonyos-ai-workspace/cli`
- npm 包发布到 cnpmjs 镜像
- 加 `tools/check-network.sh`：自动探测 raw.githubusercontent.com 可达性，不通则切镜像
- curl 一行装保留作 demo / 国内备选
- npm 包加 SHA-256 checksum + `npm provenance` 签名

### 体验层 — 上手友好度（建议做）

#### **X-1：根目录大扫除**（原 P2-2）
**为什么真值得**：第一印象是真问题。`PLAN.md` / `OPEN-SOURCE-STRATEGY.md` / `RESEARCH-NOTES.md` 是项目内部元数据，对首次 clone 的鸿蒙开发者无价值反占视野。对比池里没有第二家在根目录晾这些。

**具体动作**：
- 挪走：`PLAN.md` / `OPEN-SOURCE-STRATEGY.md` / `RESEARCH-NOTES.md` → `docs/internal/`
- 保留：`CHANGELOG.md` 根级
- 新增：`QUICKSTART.md` 给 30 秒上手用户

#### **X-2：README 顶部 30 秒 quickstart + 截图**（原 P2-5）
**为什么真值得**：冷启动测试角色 1（鸿蒙新手）卡在"不知道 Claude Code 是什么"。低成本高收益。

**具体动作**：
- README 顶部加"如果你不知道 Claude Code 是什么，先看这里"段落
- 1 张截图（不需要 GIF——维护成本高、容易过期）

#### **X-3：00-09 单文件目录有条件合并**（原 P2-3 修订）
**v2 修订**：加条件——

如果 02、03、04、05、06、08、09 这些目录**未来确实会扩展**（已规划填充内容），保留目录结构不动；如果**确认不会扩展**，合并为 `docs/topics/{...}.md`。

**判断方法**：检查 PLAN.md 是否有"未来填充哪些目录"的规划。如无，合并；如有，加 `WIP.md` 占位标记。

### 附录建议 — 留着也行（可选）

#### **A-1：llms.txt 处置**（原 P1-5 降级）
**v2 重新审视**：调研显示 llms.txt 单边发布、Google 表态没 LLM 在用，但**留着无害也无收益**。删 vs 留的实际差异是根目录 1 个文件。

**结论**：作为**主动决策点**而非必改项——如果 A 未来要上 docs 站，挪到 `docs/llms.txt`；如果不做 docs 站，按"根目录大扫除"原则一起删。本身不是独立优化项。

### Backlog — 独立项目级工作量（不进当前计划）

#### **B-1：upstream-docs 13524 md 改为 RAG MCP**（原 P2-1 降级）
**v2 重新审视**：做一个完整 RAG MCP server 是独立项目。做一半反而是技术债。当前"可选拉取"姿态不算坏。

**结论**：明确标"未来 explore 项"，不进任何当前计划。值得做时**单独建仓库** `mcp-harmonyos-docs`。

#### **B-2：设备动作 MCP**（原 P2-4 降级）
**v2 重新审视**：
- 调真机已不是 Top 痛点（DevEco 5+ 改善）
- G 早于鸿蒙 6 停更，需要 fork 重写
- 工作量大，跟用户头号痛点关联弱

**结论**：明确标"非必做"。先在鸿蒙 6 真机跑一遍 G 的 16 个 tool 看 pass rate；< 80% 就放弃，等真有用户提需求再说。

## 9. v1 → v2 修订点总结

1. **重新分级**：从"P0/P1/P2 工作量优先级"改为"C/G/E/X/A/B 价值类型分层"——更诚实地反映"谁直接对应用户痛点"
2. **C-1（ArkTS 规范库）从 P1 升级为核心层** — 直接对应鸿蒙开发者 #1 痛点
3. **E-1（AGENTS.md）从 P0 降级为扩展层** — 它扩展用户面但不解决鸿蒙痛点；分级要诚实
4. **E-1 表述修订**：不强行 symlink，是"AGENTS.md 通用宪法 + CLAUDE.md 引用 + skill 索引段"
5. **P0-7 + P1-1 合并为 E-4** — 同一件事（让首装可靠）的两面
6. **A-1（llms.txt）降级为附录决策点** — 删 vs 留实际差异小
7. **B-1（RAG MCP）和 B-2（设备动作 MCP）降级为 Backlog** — 独立项目级工作量
8. **X-3（单文件目录合并）加条件** — 如有扩展计划则保留
9. **新增第 6 节"按用户痛点关联度反向校验"** — 证明每条建议不是为做而做
