## 1. 八向定位图

A 是“规则工作区 + Claude 强钩子 + 有限多工具同步”；B 是“npm 安装器 + 大资料/模板包”；C 是“单源多工具配置导出器”；D/E 是薄 Skill/语言专项；F 是官方 SDK 垂直包；G 是动作型 MCP；H 是静态资料。A 与 B 在规则+钩子重合，A 与 C 在多工具重合。空白市场是“规则+动作 MCP+小体积安装器”和“官方/三方 SDK 分层 Skill 市场”。

| 项目 | 形态轴 | 完整度轴 | 位置判断 |
| --- | --- | --- | --- |
| A | 工作区 + curl 安装 | 规则+钩子+多工具同步 | 工程化最全，但认知负担也最高 |
| B | npm 安装器 + Skill 包 | 规则+钩子+脚手架 | 上手最像产品，默认装入内容重 |
| C | 单文件规则 + fan-out | 规则+多工具同步 | 多工具覆盖最广，无校验闭环 |
| D | Skill 包 | 仅规则 | Claude Code 薄包，重官方引用 |

## 3. A 的差异化优势与短板

只有 A 在做且有实际价值：1）OHPM 包名黑/白名单 + `ohpm view` 校验，直接打 AI 虚构依赖痛点；2）`AGC-RJ-001..020` 稳定 ID，可把上架拒因、review finding、扫描输出串起来；3）OpenHarmony 文档镜像 bootstrap，适合断网或官网搜索差时离线检索；4）PostToolUse 结果写入 `.claude/.harmonyos-last-scan.txt`，形成 AI 下一轮可读反馈；5）hook fixture 回归，至少能防扫描器退化。

A 不如别人深：B 的 `npx` 安装、PowerShell hook、starter-kit 和提审/产品质量模板更像发行包；C 的 11+ fan-out 更广；E 的 ArkTS spec/迁移章节搜索更适合编译错误根因；F 的百度地图 SDK 本地 API refs 是垂直场景的标准答案；G 的 hdc 动作工具补了 A 只读规则库的空白。

所有人都做、A 无明显差异：ArkTS 禁 `any`/解构/`delete`、Stage/ArkUI/状态装饰器、`@kit`/`@ohos` 迁移、构建命令和“查官方文档不要瞎编”。这些应保留在核心规则，但不值得在 README 当差异化反复强调。

## 4. A 的过度设计/冗余/精简点

1. `README.md`：把“一图流”和“内置内容”压缩，把长解释移到 `docs/USAGE-GUIDE.md`。痛点是新手 30 秒读不到命令；风险是卖点变弱。
2. `README.md`：把 Cursor/Copilot、镜像、卸载三条 `curl -fsSL ...` 占位替换为完整 URL 或环境变量。对比 B/C 都给可复制命令；风险低。
3. `PLAN.md` 与 `docs/PLAN.md`：保留一个源，另一个删掉或改成短链接。重复 507 行会让维护决策分叉；风险是旧链接失效。
4. `docs/REVIEW-2026-05-*.md`：移到 `docs/archive/reviews/`，不要和正式指南平级。clone 首屏被内部复盘淹没；风险是贡献者少一层上下文。
5. `tools/install.sh`：安装时补拉 `tools/data/ohpm-blacklist.txt` 和 whitelist，或把 README “~25 项”改成内联 11 项。现在外部数据不会进入 app；风险低。
6. `.cursor/rules/harmonyos.mdc`：要么提交生成物，要么 install 时先下载 `tools/generate-ai-configs.sh` 本地生成。当前 README 声称支持 Cursor，但默认远程文件未跟踪；风险中。
7. `.mcp.json`：从裸 `mcp-harmonyos` 改为 `npx -y mcp-harmonyos@latest` 或在 installer 中检测安装。fresh clone 更稳；风险是每次启动依赖网络。
8. `CLAUDE.md`：继续瘦到 250-300 行，把低频构建/签名细节放进 skill references。风险是 Claude 首轮少读一部分边角知识。
9. `.DS_Store`：从 `tools/`、根目录跟踪文件删除并加入 `.gitignore`。无用户价值；风险近零。

## 6. A 的“不做”清单评估

11 工具 fan-out：合理。C 证明可做，但维护的是重复文本，不解决鸿蒙编译痛点。A 应补一句“通过 AGENTS.md 覆盖兼容工具，显式生成只做高频工具”。

16 业务模板：基本合理。B 的 starter-kit 对新手有价值，但业务模板 6 个月老化是真风险；A 应改为 3-4 个稳定 recipe，而非完整脚手架。

自家 LLM 微调：合理。对比池无人做；现阶段 RAG/规则/校验 ROI 更高。

自家 ArkTS 解析器：合理。A 的 grep hook 应定位为快扫，最终仍调用 `hvigorw codeLinter`；不要走自研 parser。

项目脚手架：部分合理。DevEco 新建项目足够，但 F 的空工程 skill 对远程 AI 有用。建议不做生成器，做“DevEco 模板后检查器”。

Slack 群运营：合理。B 放 QQ 群是社区入口，不是产品核心能力。A 当前应先修安装闭环。

## 7. 冷启动测试（A/B）

A 新手 30 秒会卡在：先要懂 Claude/Codex/Cursor/Copilot 是什么；“用法 A/B”清楚但上方卖点太多；`curl ... --targets` 等变体命令不能复制；不知道 install 后是否会改自己的仓库哪些文件；`.mcp.json` 还要不要先 npm install。

A 的 AI 视角会先读 `AGENTS.md` 顶部和 `CLAUDE.md` hook 段，但 README 同时说人类不必读 CLAUDE、又给大量 AI 精华规则，入口层级偏多。

B 新手 30 秒更容易执行：`npx harmonyos-skills-pack` 明确，也有 uninstall。困惑是默认同时写 `.claude/.github/.codex` 且超过 300 个文件，用户未必想污染项目；“2025 激励”在 2026 语境下需要更新。

B 的 AI 视角路由更短：`CLAUDE.md` 直接指向 `.claude/skills/...`。但它缺 A 这种 Edit 后强阻断和 OHPM 校验闭环。

结论：B 更“立刻能装”，A 更“装后更能纠错”。A 当前最大冷启动问题不是能力弱，而是 README 与 installer 的可复制性、真实文件清单、Cursor/MCP 可用性没完全闭环。
